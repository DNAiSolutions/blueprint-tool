
## What I’m seeing (mapped to your 5 issues)

### 1) Multi-select scrolling is broken
**Why it’s happening in the code**
- `MultiSelectCheckbox` uses Radix `ScrollArea`.
- Our `ScrollArea` implementation uses a `Viewport` with `className="h-full"`.
- In `MultiSelectCheckbox` we set **only `maxHeight`** on the ScrollArea root, not an actual `height`, so `h-full` has no height to fill. Result: the viewport can collapse and you can’t scroll (especially on smaller panels).

**Fix**
- Ensure the options container has a real, constrained height.
- Easiest + most reliable: replace the Radix `ScrollArea` usage inside `MultiSelectCheckbox` with a plain div:
  - `className="max-h-64 overflow-y-auto space-y-1"`
- Alternatively, keep `ScrollArea` but set `style={{ height: maxHeight }}` (not just maxHeight), and/or adjust `ScrollAreaPrimitive.Viewport` to use `max-h-full` instead of `h-full`.

Outcome: You can always scroll through all options on lead sources, lead handling, qualification, etc.

### 2) Selecting options doesn’t create nodes (happening across option-based questions)
**Why it’s happening**
There are actually **two separate problems** here:

#### A) The session is often not loaded, so node creation gets ignored
- In `Canvas.tsx`, `handleNodeCreate` immediately returns if `!currentSession`.
- In `useSession.tsx`, sessions are stored only in React state (memory). If you refresh, open a direct link, or come back later, `sessions` is empty.
- `loadSession(sessionId)` just searches `sessions` state; if it’s empty, `currentSession` stays null → nothing can be added.

This explains the screenshot where you answered questions, but the canvas stays empty.

#### B) Some “option questions” have no node logic attached
- Example: intake mapping questions generated by `generateIntakeMappingQuestions()` have **no `nodeCreation`** defined, and `QuestionPanel` has no special-case handler for those IDs.
- So even when the session exists, those mapping questions don’t update the canvas.

**Fix**
1) Add **session persistence** so `currentSession` is always available:
   - Persist `sessions` + `currentSessionId` to localStorage (and restore on app load).
   - Update `loadSession()` to load from localStorage if not found in memory.
   - If a sessionId exists in the URL but not found, we should either:
     - restore it from storage, or
     - create a “placeholder session” with that id and continue (prefer restore).

2) Add “node dispatch” for all option-based questions, including dynamic ones:
   - Ensure multi-select questions with `nodeCreation.createPerSelection` always create nodes (already coded for q4 and q_intake_methods).
   - Add special-case handling for IDs starting with `q_intake_map_` (see next section).

Outcome: When you click “Answer & Continue” on any option-based question that is supposed to create/update/relate nodes, the canvas updates immediately.

### 3) Intake mapping: can’t say “TikTok leads come via SMS, Google leads come via form”
**What’s missing**
- We do generate mapping questions (`q_intake_map_${source}`), but currently they:
  - collect the answer
  - do not update any nodes or connections

**Fix (within the system you already approved)**
- Implement handling for mapping questions:
  - When question id matches `q_intake_map_(.+)`, parse the `sourceValue` from the id.
  - For each selected intake method, call:
    - `onNodeCreate('intake-connection', { intakeId, leadSourceId: sourceValue })`
- This matches what `Canvas.tsx` already supports:
  - it finds the intake node by `sourceId === intakeId`
  - then appends `leadSourceId` into `sourceConnections`

Outcome: You can specify different intake methods per lead source, and connectors draw accordingly.

### 4) Canvas doesn’t build at all
This is the same root cause as **2A** (session not loaded / no persistence), plus one more UX issue:

**UX issue**
- Even if session eventually loads, the user can answer questions before it’s available, and those node creations are dropped.

**Fix**
- Add a “session loading/ready” state on the Canvas page:
  - If `currentSession` isn’t ready, show a small “Loading session…” state and temporarily disable `Answer & Continue`.
  - Or queue `onNodeCreate` events until session becomes available, then replay them.

Outcome: You’ll never have a “blank canvas even though I answered questions” situation.

### 5) Manual node-to-node connections (critical for real calls)
**What’s missing**
- Right now, nodes can be selected for highlight, but there’s no “connect mode” and no UI for creating connections.
- Connections are stored in `SessionNode.connections` (string[] of node IDs), and connectors are rendered from that array.

**Fix**
Implement a simple “Connector Mode”:
- Add a toggle in the Canvas action bar (near Funnel View / Export PDF):
  - “Connect” (on/off)
- Behavior when Connect Mode is ON:
  - First click selects “From” node (show a small prompt like “Select a target node…”)
  - Second click selects “To” node and creates a directed connection:
    - `updateNode(fromId, { connections: [...existing, toId] })`
  - Prevent duplicate edges.
- Add a quick way to remove a connection:
  - Option A (simple): if a node is selected, show a “Clear Connections” button
  - Option B (better): allow clicking a connector line to remove (requires pointer events on SVG; can be done, but a bit more work)

Outcome: Reps can map messy real-world flows live, not only the guided “auto-built” structure.
